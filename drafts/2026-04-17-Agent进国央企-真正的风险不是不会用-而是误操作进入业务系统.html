Agent 进国央企，真正的风险不是不会用，而是“误操作进入业务系统”

作者：余炜勋
来源：知识库 + 延伸

如果把前两篇连起来看，其实逻辑已经很清楚了。

第一篇讲的是：为什么很多 AI 一体机会在国央企机房里吃灰。
第二篇讲的是：为什么很多国央企不是没预算，而是没把 AI 放进业务闭环。

那第三篇，问题就自然来了：

如果在闭环还没补上、组织能力还没跟上的情况下，国央企又开始大规模上 Agent，会发生什么？

我的判断很直接：

真正的风险，不是 Agent 不会用，而是它一旦真的开始“会做事”，风险就会从“吃灰”升级成“误操作进入业务系统”。

这件事比很多人想得都严重。

因为传统对话式大模型，哪怕胡说八道，很多时候也只是“内容错误”。但 Agent 一旦具备调系统、调接口、发指令、串流程、触发动作的能力，风险性质就完全变了。

它不再只是一个会说错话的助手，而可能变成一个会做错事的执行者。

一、为什么 Agent 比普通大模型更容易让人上头？

因为它更像“真的能干活”。

过去很多单位看大模型，最大的质疑就是：

  • • 会写，但不会做
  • • 会答，但不会办
  • • 会聊天，但不进流程

而 Agent 恰好击中了这个痛点。

它给人的想象空间太大了：

  • • 可以自动调用知识库
  • • 可以读取流程节点
  • • 可以串多个系统
  • • 可以自动填表、发邮件、派单、查数据
  • • 可以像“数字员工”一样跑任务

这套叙事对国央企是非常有诱惑力的。

因为很多单位现在最焦虑的，不是有没有模型，而是模型能不能真正进业务。

所以一听 Agent 能“执行”，很多人会天然觉得：

这次终于不是演示工具了，而是真的能上岗了。

问题就在这里。

Agent 最大的吸引力，恰恰也是它最大的风险来源。

二、为什么说 Agent 一进系统，风险就升级了？

因为一旦进入执行层，它面对的就不再只是语义问题，而是权限问题、流程问题和责任问题。

你可以把普通大模型理解成“会说话的分析师”，它再怎么出错，很多时候还是停留在建议层。

但 Agent 更像什么？

更像一个被授予了一部分操作权限的新人。

而且这个新人有几个非常危险的特征：

  • • 理解能力并不稳定
  • • 对上下文极其敏感
  • • 可能过度自信地执行
  • • 遇到边界条件时容易误判
  • • 会把“看起来合理”当成“真的可以做”

如果这种能力只停留在测试环境，问题不大。

但如果它被接进真实业务系统，风险马上就不一样了。

它可能不是答错一句话，而是：

  • • 发错一封邮件
  • • 触发一次错误审批
  • • 调错一个接口
  • • 删除一批不该删的数据
  • • 把内部材料推送给错误对象
  • • 把业务流程推进到不该推进的节点

这时候，错误就不再是“AI 幻觉”，而是实打实的业务事故。

三、国央企为什么特别容易在 Agent 这件事上踩坑？

不是因为国央企更笨，而是因为它的业务环境天然更复杂、更重责任、更重权限边界。

至少有四个特点，决定了 Agent 在国央企场景里的风险会被放大。

1. 系统多，流程长，权限边界复杂

很多国央企不是只有一个系统，而是：

  • • 办公系统一套
  • • 业务系统一套
  • • 财务系统一套
  • • 采购系统一套
  • • 合同系统一套
  • • 档案系统一套
  • • 各类条线系统再来一套

系统多本身还不是最麻烦的。

更麻烦的是，权限和流程往往是按组织层级、岗位职责、制度要求严密切出来的。

在这种环境里，Agent 真正难的不是“能不能连上系统”，而是：

它有没有能力正确理解哪些动作可以做，哪些动作必须停，哪些动作必须由人二次确认。

如果这一层做不好，系统接得越深，事故风险越大。

2. 责任机制是刚性的，但 Agent 的决策逻辑是概率性的

国央企业务里，很多关键动作都不是“建议性动作”，而是责任性动作。

比如：

  • • 公文流转
  • • 合同审批
  • • 采购申请
  • • 供应商准入
  • • 预算调整
  • • 数据报送
  • • 风险预警

这些动作一旦错了，不是“改一下就行”，而是可能直接牵涉：

  • • 审计问题
  • • 合规问题
  • • 责任追溯问题
  • • 业务延误问题
  • • 信息安全问题

问题在于，Agent 的底层并不是一个确定性规则系统，而是一个高度依赖概率判断和上下文理解的智能体。

这就形成了一个天然冲突：

国央企要求的是可追责、可解释、可审计；Agent 天然擅长的却是概率推断、语义联想和柔性执行。

如果中间没有特别强的约束层，它就很容易出事。

3. 很多组织还没准备好“人机共责”的新机制

传统信息系统里，责任很清楚：

  • • 人点的
  • • 系统记的
  • • 审批链有痕迹
  • • 出事能回放

但 Agent 一旦加入，问题会复杂很多。

比如一个动作到底算谁做的？

  • • 是操作人发起的？
  • • 是 Agent 判断的？
  • • 是系统接口执行的？
  • • 是产品规则设计导致的？
  • • 还是厂商模型能力不稳定造成的？

如果这套责任链没有提前设计，Agent 看起来越智能，后面追责时就越混乱。

4. 很多单位还没有把 Agent 当“高风险执行系统”来管

现在市场上最危险的一种倾向就是：

很多人把 Agent 当成“更聪明的助手”，但它在真实业务里更接近“高风险自动化执行系统”。

这两者的管理要求完全不是一个等级。

助手可以建议，错了改一下。
执行系统一旦进流程、调用接口、改变状态，就必须纳入更严格的：

  • • 权限控制
  • • 审批拦截
  • • 沙箱隔离
  • • 日志留痕
  • • 回滚机制
  • • 人工兜底

如果这些没有先建起来，Agent 越聪明，反而越危险。

四、为什么很多单位现在上 Agent，实际上是在“跳级”？

这个问题特别关键。

很多国央企现在连普通大模型进入业务闭环这件事都还没跑顺。

也就是说，以下这些基础问题其实都还没真正解决：

  • • 数据治理做没做好
  • • 场景边界清不清楚
  • • 权限控制稳不稳定
  • • 责任链条明不明确
  • • 持续运营团队有没有
  • • 业务 owner 在不在场

在这些问题没解决之前，直接上 Agent，本质上就是在跳级。

像什么？

像一个单位连基础内控都没补好，就急着上全自动财务机器人；连知识库都没整理明白，就开始做跨系统自动执行。

不是不能做，而是极容易出大问题。

所以很多时候，Agent 项目最值得问的，不是“能不能做”，而是：

现在这个组织，真的配得上一个有执行能力的智能体吗？

这话听起来有点狠，但现实就是这样。

五、国央企真要上 Agent，至少要先补哪几层？

如果真想让 Agent 进业务，又不想把它变成事故制造机，我觉得至少要先补五层。

1. 场景边界层

先明确它到底能干什么，不能干什么。

不要一上来就让它碰高风险动作。先从低风险、可回放、可撤回、可审计的场景开始。

2. 权限控制层

Agent 不应该拿到模糊权限，只能拿到被严格约束、最小化、可收回的权限集合。

3. 人工确认层

任何影响业务状态、资金流、审批结果、对外发送的信息，都必须有人类兜底，不要幻想“全自动”一步到位。

4. 日志审计层

它每一步做了什么、基于什么上下文、调用了什么工具、触发了什么结果，都必须留痕。

5. 回滚与熔断层

一旦判断异常，系统必须能停、能撤、能切回人工流程。

如果这五层不先补，Agent 越深入系统，组织就越暴露在不可控风险里。

六、最后一句判断

Agent 进国央企，真正的风险从来不是“不会用”，而是它一旦真的开始进入系统、串联流程、触发动作，很多单位其实还没有准备好承接由此带来的权限、责任和事故风险。

所以真正该警惕的，不是 Agent 不够聪明，而是组织高估了自己对 Agent 的控制能力。

在闭环没补上、边界没划清、兜底机制没建好的情况下，Agent 不是效率工具，更可能是一个披着智能外衣的高风险执行入口。

今天很多人还在担心 Agent 会不会“没用”，但更该担心的是：

它一旦开始“有用”，会不会在没人看清风险的时候，把错误真正推进业务系统里。

这件事，不是技术乐观主义能解决的，而是组织治理能力要先跟上。
