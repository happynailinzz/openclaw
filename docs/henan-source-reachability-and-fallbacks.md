# 河南关键源站可达性排查 + 备用抓取方案

更新时间：2026-03-08

## 结论先说

当前“两会×河南×产业监控”的质量瓶颈，不是总结能力，而是关键河南源站的可达性不稳定。

本机实测结果：
- `henan.gov.cn`：403
- `fgw.henan.gov.cn`：403
- `m.jyt.henan.gov.cn`：403
- `hca.miit.gov.cn`：200
- `www.zhengzhou.gov.cn`：200
- `www.zzu.edu.cn`：200
- `henb.nea.gov.cn`：200
- `www.gov.cn`：200

这说明：
1. 省政府 / 发改委 / 教育移动站这批河南关键源，在当前服务器出口上被 CDN/WAF 直接拦截。
2. 工信系统、能源监管、高校、国务院源当前可稳定使用。
3. 监控若继续依赖 `henan.gov.cn` / `fgw.henan.gov.cn` 直抓，质量会持续波动。

2. `https://henb.nea.gov.cn/...`

- 价值：高
- 适合用途：能源数字化、新型电力系统、电力市场、监管动态

3. `https://www.zzu.edu.cn/...`
- 状态：200
- 价值：中高
- 适合用途：产教融合、AI人才培养、成果转化、区域创新

4. `https://www.gov.cn/...`
- 状态：200
- 价值：高
- 适合用途：国家层政策基线、政府工作报告、政策新表述

5. `https://www.zhengzhou.gov.cn/`
- 状态：200
- 价值：中
- 问题：首页可达，但正文线索提取能力有限，需要更具体栏目页/专题页

### B. 当前被拦截

1. `https://www.henan.gov.cn/...`
- 状态：403
- 观察：CDN/WAF 直接拒绝，响应头包含 `x-via`、`Ws-Action: com`
- 判断：不是解析问题，不是单次偶发，是源站防护策略/出口限制

2. `https://fgw.henan.gov.cn/...`
- 状态：403
- 判断同上

3. `http://m.jyt.henan.gov.cn/...`
- 状态：403
- 判断：同属当前出口不友好源

## 失败模式判断

当前失败更像“出口/IP/风控策略不兼容”，不是简单的 UA 问题。

证据：
- `curl -I -L -A 'Mozilla/5.0'` 仍是 403
- Python urllib 也同样 403
- 多个河南省级站点一致表现，且返回头风格相近

因此，不建议继续在当前服务器上对这些站点做无意义重试。

## 备用抓取方案（按优先级）

### 方案 1：保留直抓白名单源，作为主链路

主链路保留这些源：
- `hca.miit.gov.cn`
- `henb.nea.gov.cn`
- `www.gov.cn`
- `www.zzu.edu.cn`
- `www.zhengzhou.gov.cn`（需细化到栏目/专题）

优点：
- 稳
- 可自动化
- 权威度够

缺点：
- 河南省政府/发改委这类核心本地源缺位

### 方案 2：对 403 源改成“搜索命中 + 外部镜像/转载 + 占位页”模式

适用源：
- `henan.gov.cn`
- `fgw.henan.gov.cn`
- `m.jyt.henan.gov.cn`

做法：
1. 先用搜索/聚合链路确认是否有新文章 URL。
2. 若正文直抓失败：
   - 保留原始 URL
   - 记录标题、日期、栏目、摘要（若可从搜索结果拿到）
   - 标记为“原站受限，待补正文”
3. 同时寻找：
   - 官方转载页
   - 权威媒体转述页
   - 相关单位二次发布页

优点：
- 不会因为 403 直接断档
- 仍可保留情报线索和追踪能力

缺点：
- 正文完整度不稳定

### 方案 3：对关键 403 源建立人工补链清单

适用于高价值、必须跟踪的栏目：
- 河南省政府政策/专题页
- 河南省发改委两会/产业/项目页
- 河南教育系统两会与产教融合页面

做法：
- 只保留少量关键栏目，避免全站瞎抓
- 将其列为“高价值但受限源”
- 在监控输出中单独列出：
  - 是否发现新链接
  - 是否拿到正文
  - 是否需要人工补链

优点：
- 重点突出
- 不浪费抓取资源

## 对监控脚本/任务的直接建议

### 一、源头分层

主监控源：
- 国务院 / 部委（可直抓）
- 河南通信管理局
- 河南能源监管办
- 郑州大学
- 郑州市政府具体专题/栏目

受限源：
- 河南省政府
- 河南发改委
- 河南教育移动站

### 二、输出时显式标注抓取状态

每条线索建议附一个状态：
- `正文已拿到`
- `仅拿到标题/摘要`
- `原站 403，待补链`

这样用户一眼就能看出哪些是硬情报，哪些只是占位线索。

### 三、不要对 403 源做高频重试

建议：
- 每轮最多 1 次直抓
- 若失败，直接切 fallback
- 连续多轮失败时，在简报中合并为一句，而不是反复刷失败日志

## 已落地的执行方案

### 1. 浏览器代抓路由（已接入）
- 路由配置：`config/henan-source-routing.json`
- 执行脚本：`scripts/henan-browser-fetch.py`
- 当前已接入：`m.jyt.henan.gov.cn`
- 当前行为：走本机已恢复的 Chrome CDP（`127.0.0.1:9222`）打开页面，不再走服务器直抓。

### 2. 403 源 fallback 规则（已接入）
- 当前已接入：
  - `www.henan.gov.cn`
  - `fgw.henan.gov.cn`
- 当前行为：直接返回 `原站403，待补链`，并附带 fallback 策略，不再对原站做无意义硬撞。

## 建议落地顺序

1. 先把监控源分成“可直抓白名单”和“403 受限名单”
2. 再把输出模板改成带抓取状态的决策简报
3. 最后再补搜索/镜像/转载的备用链路

## 当前最现实的结论

在这台服务器上：
- 不能把 `henan.gov.cn` / `fgw.henan.gov.cn` 当作稳定直抓源
- 可以把它们当作“高价值受限源”保留追踪
- `m.jyt.henan.gov.cn` 已验证可走浏览器代抓链路
- 真正稳定的自动监控，当前应建立在“可直抓白名单 + 浏览器代抓 + fallback 占位”三层结构上
