# CRON-RULES.md

## 设计原则

1. 精确定时任务才用 cron；可漂移的巡检放 heartbeat。
2. 成功静默，失败告警（避免刷屏）。
3. 每个 cron 必须定义：
   - 任务目标
   - 输出对象
   - 失败处理
   - 重试策略

## 命名规范

- 分类:动作:时间
- 示例：
  - digest:ai-hot:0900
  - monitor:github:0900
  - healthcheck:update:0400

## 失败处理规范

- 首次失败：记录日志 + 轻提醒。
- 连续失败2次：升级提醒（标记「需人工介入」）。
- 连续失败3次：停止自动重试，等待人工确认后恢复。

## 输出规范

- 成功：默认不主动消息（或汇总到每日固定时段）。
- 失败：输出三段：
  - 【失败任务】任务名 + 时间
  - 【失败原因】错误摘要
  - 【建议动作】最多3条

## 推荐任务分层

- 高频轻任务（每1-3小时）：
  - 链路连通、抓取可用性、账号登录态
- 中频任务（每天固定时点）：
  - 热点简报、项目更新
- 低频重任务（每周）：
  - 安全审计、记忆维护、配置体检
