# OpenClaw 生产化 SOP v1.0（东城老饕定制）

更新时间：2026-02-27  
适用环境：当前 OpenClaw 主实例（workspace）

---

## 0. 目标

把现有能力固化成“稳定可复用”的生产流程，重点保证：

1. 内容生产不断档（选题/写作/改稿/发布）
2. 入库一致（Notion 同库、同标准）
3. 报警可追踪（心跳 + cron）
4. 出问题可回滚（脚本化巡检与恢复）

---

## 1. 固定约束（必须遵守）

### 1.1 信息源优先级（公众号相关任务）

1. `/opt/openclaw/knowledge/secendME`
2. `MEMORY.md` 与历史会话
3. 外部资料

输出必须标注来源类别：知识库 / 延伸 / 外部。

### 1.2 搜索策略

- 默认优先 Tavily 系列能力
- 不主动使用 Brave（除非用户明确要求）

### 1.3 Reddit 策略

- 固定为：Tavily -> Exa
- 暂不启用代理

### 1.4 敏感信息

- 严禁在日志、文档、提交中写入明文 key/token
- 配置走环境变量或受控配置文件

---

## 2. 内容生产标准流水线

## 2.1 主链路

选题 -> 草稿 -> 审校 -> HTML（复制友好）-> Notion 入库 -> Telegram 交付

## 2.2 HTML 标准（硬要求）

- 必须复制友好：`ul/ol/li` + `pre/code`
- 优先可复制和可二次编辑，不追求花哨样式

## 2.3 产物路径规范

建议统一放在：

- `articles/YYYY-MM-DD-主题.md`
- `articles/YYYY-MM-DD-主题.html`
- `articles/YYYY-MM-DD-主题-复制友好.html`

---

## 3. Notion 入库规范

## 3.1 默认目标库

- 文章选题草稿箱（历史主库）
- 微信公众号文章库（外部文章收藏与正文沉淀）

## 3.2 入库字段最低要求

- 标题
- 原文链接
- 来源标记
- 阅读状态（未读/阅读中/已读）
- 收藏日期 / 发布日期（能取到就写）
- 正文（可用时写入）
- 备注（标注抓取链路与不确定性）

## 3.3 同页覆盖规则

对于“迭代型草稿”：

- 保持同一页 ID
- 先清空旧 block 再写新内容
- 更新后回传页面链接

---

## 4. X 文章抓取回退链路（标准）

按顺序尝试，直到拿到正文：

1. 直连 x.com
2. `vxtwitter/fxtwitter + r.jina.ai`
3. 标记“抓取受限”，先入库链接，等待补源

入库备注必须写清：

- 使用了哪条抓取链路
- 是否全文
- 是否存在排版噪声

---

## 5. 运行巡检与恢复

## 5.1 日常巡检（建议每天）

1. `scripts/verify-openclaw.sh`
2. `openclaw status`
3. cron 最近执行状态
4. 关键渠道登录态（X / 小红书）

## 5.2 故障恢复（按优先级）

1. 配置与依赖检查
2. 运行 `scripts/bootstrap-openclaw.sh`
3. 重跑 `scripts/verify-openclaw.sh`
4. 单链路回归测试（Notion / Telegram / XHS）

---

## 6. 心跳与 Cron 分工

## 6.1 Heartbeat（轻量、合并、低打扰）

检查项：

- AI 热点选题
- 政企数字化热点
- 海外中文新闻
- 进行中任务状态

无变化：只回 `HEARTBEAT_OK`

## 6.2 Cron（精确触发）

- 用于固定时点任务（如 09:00 监控）
- 关键任务失败要有告警
- 非关键任务合并汇报，避免消息轰炸

---

## 7. 质量门禁（发布前 60 秒检查）

发布前逐条确认：

1. 内容事实是否可追溯
2. 来源标注是否完整（知识库/延伸/外部）
3. HTML 是否可复制（列表/代码块）
4. Notion 页面是否更新成功
5. Telegram 是否发送成功
6. 如果失败，是否给出可执行回退方案

---

## 8. 成本与模型策略

- 轻任务：快模型
- 复杂写作/系统设计：强模型
- 长流程优先“先结构后生成”，减少返工 token
- 定期复盘：按周看一次高消耗任务类型

---

## 9. 版本管理

- 每次流程升级都要：
  1) 更新此 SOP
  2) 提交 git
  3) 用一句话写明“为什么改”
