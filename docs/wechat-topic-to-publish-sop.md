# 微信公众号：选题到发布 SOP（按当前工作流优化）

适用对象：东城老饕当前内容工作流  
核心原则：**知识库优先，快发优先，固定版式，少打断，一次跑完**。

---

## 0. 总原则

### 信息源优先级（固定）

1. **Obsidian 知识库**：`/opt/openclaw/knowledge/secendME`
2. **本地记忆 / MEMORY / 历史文章**
3. **外部资料 / 热点 / 政策 / 新闻**

### 产出原则（固定）

- 默认作者：`余炜勋`
- 默认目标：**公众号可发布成稿**，不是普通草稿
- 默认输出：优先直接走 **refined wechat HTML**
- 默认排版：严格沿用既有固定版式
  - 居中下划线主标题
  - 蓝色副标题 / 作者
  - 蓝底白字二级标题
  - 左侧蓝线引用块
- 避免：
  - 单独一个“。”占一行
  - 零碎残句单独占一行
  - 泛杂志风、过度卡片风

### 执行原则（固定）

- 默认一次性跑完，不逐步确认
- 仅以下情况中断：
  1. 高风险外发 / 覆盖 / 删除
  2. 关键信息缺失
  3. 账号 / 发布权限 / 封面素材异常

## 1. 选题阶段：先判断“值不值得写”

### 1.1 选题入口分三类

#### A. 存量知识驱动
适合：深度稿、判断稿、行业方法稿

操作：
- 先查 Obsidian 知识库是否已有相关笔记
- 再查 MEMORY / 历史文章中是否已有可复用观点
- 最后再看是否需要外部补充

适用主题：
- 政企数字化
- 能源数字化 / 源网荷储
- 信创国产化
- 数据治理 / 数据要素
- 智慧教育 / 产教融合

#### B. 热点触发驱动
适合：快反稿、政策稿、窗口期稿

推荐组合：
- `daily-trending`：看当天平台热榜 / 大众热度
- 外部政策 / 新闻抓取：看真正与你业务相关的新增信号
- `content-brainstorm`：把热点转成可写选题

判断标准：
- 能不能从“新闻”转成“项目机会 / 业务窗口 / 交付逻辑”
- 能不能挂回寰曜数能既有业务版图
- 能不能给出行业判断，不只是复述新闻

#### C. 任务型选题
适合：指定方向、指定地区、指定政策

例如：
- 河南某地市 AI / 电力 / 数据治理机会
- 某项政策对 G 端 / B 端项目的影响
- 某行业客户能落什么项目

这类不先追热点，先做问题拆解。

## 2. 选题生成 SOP（推荐）

### Step 1：先拿“原始信号”

优先取：
- Obsidian 已有笔记
- 今日新增政策 / 新闻 / 热点
- 最近 7 天历史文章空缺位

### Step 2：用 `content-brainstorm` 做第一轮选题扩写

输出至少包含：
- 题目
- 切入角度
- 适配平台（公众号优先）
- 关键词
- 为什么值得今天写

### Step 3：人工做一轮“业务化筛选”

只保留符合以下条件的题：
- 能连接政策 → 场景 → 项目 → 交付
- 能体现寰曜数能业务版图
- 能输出判断，不只是信息搬运
- 标题有传播性，但不标题党

### Step 4：确定文章类型

把题目归入以下 5 类之一：

1. **政策解读稿**：政策变化 → 项目窗口
2. **行业判断稿**：发生了什么 → 谁受影响 → 怎么落地
3. **地方机会稿**：某省 / 某市 / 某园区 → 机会点
4. **方法路径稿**：怎么做、分几步、谁来做
5. **案例拆解稿**：谁在做、做成什么、可复制什么

## 3. 写作 SOP：从题目变成可发稿

### 3.1 标准文章结构

建议固定成这 6 段：

1. **开头破题**
   - 先讲这件事为什么值得看
   - 直接给结论，不绕

2. **信号 / 事实层**
   - 政策、新闻、数据、动作
   - 只放关键事实，不堆素材

3. **行业影响层**
   - 对谁影响最大
   - 影响的是预算、交付、招标、建设，还是组织流程

4. **项目机会层**
   - 能落哪些项目
   - 谁最容易先拿到窗口

5. **落地路径层**
   - 怎么推进
   - 从咨询、集成、产品、投融资哪个切

6. **结尾收束**
   - 再次给判断
   - 给行动建议 / 观察点

### 3.2 写作要求

必须做到：
- 结论先行
- 信息密度高
- 口语化但不油腻
- 有判断，不模棱两可
- 段落短，不拖

避免：
- 空泛 AI 腔
- 大词堆砌
- 只讲趋势，不讲项目
- 只讲技术，不讲交付

### 3.3 来源标注规则

回答 / 成稿中需分清：
- **知识库**：来自 Obsidian 哪些笔记
- **延伸**：基于既有认知做的推导
- **外部**：来自哪些新闻、政策、网页

内部草稿阶段至少在文末保留一版来源说明，方便回溯。

## 4. 配图 SOP：什么时候用 `baoyu-infographic`

### 适合调用 `baoyu-infographic` 的情况

1. 文章本身结构复杂
2. 需要做“对比 / 流程 / 路线图 / 架构图”
3. 需要一张高价值主图支撑传播
4. 文章属于：
   - 政策图解
   - 产业链图解
   - 项目路径图解
   - 能源 / 算力 / 信创结构图

### 推荐风格（按当前偏好）

优先考虑：
- `technical-schematic`
- `pop-laboratory`
- `dashboard`
- `dense-modules`
- `hierarchical-layers`

尽量少用：
- 过萌
- 过花
- 过手作感
- 过度杂志风

### 使用原则

- 信息图是加分项，不是每篇都必须有
- 快反稿优先发，图可后补
- 正式重稿 / 专题稿 / 可复用 evergreen 内容，优先补图

## 5. 发布 SOP：统一走 `scripts/publish.sh`

### 标准发布命令

#### 快发模式（默认优先）

```bash
bash scripts/publish.sh articles/你的文章.md --fast
```

如果已有封面：

```bash
bash scripts/publish.sh articles/你的文章.md --fast --cover /绝对路径/封面.png
```

#### 完整模式

```bash
bash scripts/publish.sh articles/你的文章.md
```

### `publish.sh` 已做的事

1. 处理封面图
2. Markdown 转 HTML
3. `refine_wechat_html.py` 生成 refined wechat HTML
4. 上传公众号草稿箱
5. 同步 Notion「文章选题草稿箱」

### 当前建议

- **热点 / 快反 / 占位**：默认 `--fast`
- **正式重稿 / 需要高质量封面**：走完整模式
- **如果已有合适封面**：永远优先 `--cover`

## 6. Frontmatter 规范（发稿前必查）

```md
---
title: 文章标题
author: 余炜勋
description: 文章摘要
---
```

补充建议：
- `title`：明确、可传播、不装腔
- `description`：尽量 60-120 字，能独立概括文章价值
- `author`：默认不要漏

---

## 7. 日常执行节奏（建议）

### 日更 / 快反

1. `daily-trending` 看外部热度
2. 补业务相关政策 / 新闻
3. `content-brainstorm` 扩成 3-5 个候选
4. 选 1 个最能转项目机会的题
5. 直接写稿
6. `publish.sh --fast` 入草稿箱 + 入 Notion

### 深度稿 / 专题稿

1. 先查 Obsidian
2. 查 MEMORY / 历史文章
3. 外部补证据
4. 写完整稿
5. 必要时用 `baoyu-infographic` 补主图 / 路径图
6. 走完整发布模式

## 8. 当前最优工作流（建议固定）

### 快反工作流

**daily-trending / 新闻信号**  
→ **content-brainstorm 做候选选题**  
→ **人工筛掉空题和伪热点**  
→ **写成公众号稿**  
→ **`bash scripts/publish.sh xxx.md --fast`**

### 深度工作流

**Obsidian 知识库**  
→ **历史记忆 / 旧稿复用**  
→ **补外部材料**  
→ **写成公众号正式稿**  
→ **必要时 `baoyu-infographic` 做图**  
→ **`bash scripts/publish.sh xxx.md`**

---

## 9. 不建议做的事

- 不要每篇都追热点
- 不要每篇都强行做信息图
- 不要为了“看起来像新媒体”牺牲行业判断
- 不要把小红书表达直接原封不动搬到公众号
- 不要绕开 `publish.sh` 另起一条主发布链

---

## 10. 当前推荐装机结论

已安装并建议纳入公众号工作流：
- `content-brainstorm`：用于选题策划
- `daily-trending`：用于热榜输入源
- `baoyu-infographic`：用于结构化可视化主图

当前暂不纳入公众号主发布链：
- `xhs-image-gen`

原因：
- 它更适合小红书图文生产
- 带发布脚本与 Cookie 依赖
- 先和公众号主线解耦更稳
