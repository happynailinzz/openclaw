逮虾户的春节七天（新手上手版）

一句话版：这个春节，不是我在过节，是 OpenClaw 在“上岗培训 + 连续加班 + 绩效冲刺”。 如果你也是新手，这篇我尽量写成“能照着做”的版本：你不需要一次性全懂，只要按顺序把几个关键环节跑通，就能从“能聊天”走到“能干活”。

先说结论：新手最容易踩的坑，不是不会装，而是装完不会用

很多人第一次上 OpenClaw，会卡在这三件事：

  1. 装好了，但不知道先配什么（人格、记忆、网关、渠道一团乱）
  2. 能聊几句，但没有闭环（写完东西不知道存哪、发哪、怎么追踪）
  3. 一改配置就慌（怕把系统搞挂，不敢升级、不敢做 cron）

这篇文章就是解决这三个问题：

  • 先把它跑起来
  • 再让它稳定
  • 再让它像你的同事一样长期干活

你可以先把 OpenClaw 理解成什么（新手心智模型）

别把它只当“聊天机器人”。

更准确一点，它是：

  • 一个会话系统（你和它长期协作）
  • 一个工具调度器（能调文件、命令、消息、浏览器、技能）
  • 一个工作流容器（写作、巡检、定时任务、消息推送、知识库）

所以它的价值不在“单次问答有多聪明”，而在：

  • 能不能长期记住你的约束
  • 能不能稳定调用工具
  • 能不能把结果交付到你真正用的地方（Telegram / Notion / 文件）

这也是我这七天一直在做的事：把“会说”调成“会做”。

开工前：新手建议的最小目标（别一口气上生产）

如果你刚开始用，建议第一阶段只追这 4 个目标：

  1. 能连上 OpenClaw 并稳定对话
  2. 能读写 workspace 文件
  3. 能做一次完整交付（比如写一篇稿子并发到 Telegram）
  4. 能做一次健康检查（知道系统现在是不是正常）

只要这四条通了，你就已经超过很多“装完就吃灰”的状态了。

初一：先活下来，再优雅（把系统跑起来）

第一天最朴素：先把 OpenClaw 跑起来。

我当时做的核心动作

  • 把网关拉起来
  • 把会话跑通
  • 把“我是谁、你是谁、怎么说话”这些人格文件补齐（SOUL.md / USER.md / MEMORY.md）

当时状态就像刚开工的厨房：锅有了，火也有了，但菜谱还在脑子里。

新手你可以照着做（最小检查）

先确认系统有没有活着：

openclaw status

看这几个点：

  • Gateway 是不是 reachable
  • Gateway service 是不是 running（如果你用 systemd）
  • Channels（例如 Telegram）是不是 OK
  • Sessions 有没有正常出现

如果你只想看网关服务状态：

openclaw gateway status

经验：先看状态，再动配置。很多新手一上来就改半天，结果其实服务没起来。

为什么要先写 SOUL / USER / MEMORY（不是矫情）

因为你希望它不是每次都像“第一次见你”。

这三个文件分别解决：

  • SOUL.md：它怎么说话、怎么判断、怎么拦坑
  • USER.md：你是谁、你做什么、你的偏好和约束
  • MEMORY.md：长期有效的规则和决策（比如“公众号任务先查知识库”）

这一步做完，OpenClaw 才开始像“同一个人”，而不是“随机 AI”。

初二：从“能用”到“不翻车”（补运维地基）

第二天开始补运维地基，不然后面早晚翻车。

我补了什么

  • 安全审计跑起来
  • 定时任务（cron）开起来
  • 心跳巡检节奏固定下来
  • 配置和日志开始有归档

那一刻的感受是：终于不是“临时搭子”，开始像“可值夜班”的系统了。

新手必做的 3 个“体检动作”

1）看版本和更新状态

openclaw update status

这一步很重要，因为：

  • 版本差一小步，功能/命令/兼容性可能差很多
  • 你看教程照抄时，版本不一致最容易踩坑

2）跑安全审计（先只读）

openclaw security audit --deep

你会看到一些 warn，别慌。先看懂，再决定要不要修。

我这边的典型告警就包括：

  • 反向代理信任头未配置（如果你没暴露控制台，影响没那么大）
  • 某些 denyCommands 条目命名不生效（配置项名字不精确）

重点：审计的意义不是“零告警”，而是你知道每一条告警是什么意思、风险在哪。

3）固定心跳和巡检节奏

我的策略是：

  • 活跃时段（如 08:30-23:00）定时检查
  • 没新变化就安静（避免打扰）
  • 有新热点/新任务再主动汇报

这能避免系统变成“会打扰但不产出”的电子闹钟。

初三：技能大采购，拒绝裸奔（把“会聊”变“会干活”）

把核心技能链路装齐，尤其是内容生产和检索相关。

我当时重点补的能力模块（不是乱装一堆）

  • 搜索（用于实时信息）
  • 网页抓取/阅读（把网页变成 AI 容易吃的文本）
  • 知识库（让它优先调用我的内容）
  • 文档处理（Markdown / HTML / Word / Excel 等）
  • Notion 集成（做选题与草稿入库）
  • 去 AI 味写作优化（让成稿更像人写的）

本质上做了一件事：把“会聊天”升级成“会干活”。

新手别犯的错：不要一口气装 30 个 skill

正确姿势是：按任务链路装。

比如你和我一样主要做内容/咨询交付，第一批就够了：

  • 搜索
  • 抓取
  • 知识库
  • 写作优化
  • Notion（或你常用的交付工具）

先跑通 1 条闭环，再扩展。

真的，很多人不是不会用，是把系统装成“技能博物馆”。

初四：给它灵魂，不当客服腔（风格改造）

这一趴非常关键。

同样一套工具，如果表达风格还是“企业客服公文体”，你会很快不想用它。

我做的改造方向（效果非常明显）

  • 结论先行
  • 少废话
  • 该拦坑就拦坑
  • 保留一点人味儿和幽默感
  • 不要动不动“这是个好问题”“很高兴帮助你”

效果非常直观：同样一件事，它不再像在写周报，而是像在跟你并肩做项目。

新手实操建议（很重要）

你不用一开始就写很复杂的 prompt。

直接在 SOUL.md 里写清楚这几件事就行：

  • 你要它像什么角色（同事 / 助手 / 顾问）
  • 你讨厌什么风格（废话、官腔、过度客套）
  • 它什么时候必须先问你（涉及删除、外发、改配置）
  • 它什么时候可以直接执行（读文件、整理、草拟）

这一步的收益极高，成本极低。

初五：写作流水线打通（知识库优先，不再“互联网平均值”）

这一趴是我最看重的，因为它直接决定内容是不是“你的味道”。

我固定下来的规则（后来证明很值）

公众号工作流按这个优先级走：

  1. 先查 /opt/openclaw/knowledge/secendME
  2. 再用 MEMORY.md / 会话记忆
  3. 最后才外部资料

并且输出里要标注来源类别：

  • 知识库
  • 延伸（基于你的经验与框架推导）
  • 外部

为什么这一步这么关键

因为如果不立这条规则，AI 很容易变成：

  • 到处搜点资料
  • 拼一篇看起来正确但很像别人写的东西
  • 读着顺，但没有你的判断和方法论

而知识库优先的结果是：

  • 内容更贴你的业务语境
  • 风格更稳定
  • 输出更像“长期在你团队里干活的人写的”

新手怎么开始（不需要一开始就很高级）

你哪怕先准备一个知识库目录都行，里面放：

  • 你过去写过的文章
  • 客户沟通记录（脱敏）
  • 你的常用框架
  • 行业笔记

然后告诉 OpenClaw：先用这些，再外搜。

这一步会让它“像你”，不是“像某家大模型”。

初六：Notion 入库和分发闭环（从写作动作变生产系统）

把“写完就散”改成“写完入库”。

我打通了哪些环节

  • 公众号选题与草稿默认进 Notion「文章选题草稿箱」
  • 文章可导出 HTML（方便公众号发布）
  • 可发 Telegram 做交付/预览（手机就能先看成稿）

到这一步，流程就不再是“AI 帮我写一段话”，而是：

AI 参与了一个完整的内容生产链路。

新手可以照着抄的闭环模板（内容场景）

  1. 你发需求（题目 / 方向 / 场景）
  2. OpenClaw 先查知识库
  3. 不足再补外部资料
  4. 生成初稿（标来源）
  5. 去 AI 味优化
  6. 导出 HTML
  7. 发 Telegram 预览
  8. 入库 Notion（选题/草稿管理）

这个闭环一旦跑通，你之后每篇文章都是在复制流程，而不是重造轮子。

我这边目前的真实状态（不藏问题）

  • Notion 流程规则已经定下来（默认入草稿箱）
  • 实际使用中也有过成功同步
  • 但当前 Notion API key 配置异常（环境变量里是占位内容），需要修正后恢复 API 直写稳定性

新手提醒：集成出问题很正常，关键是你要把问题定位清楚（是权限、连接、字段映射，还是 key 本身）。

初七：生产化体检，交卷（把“能跑”变“敢长期跑”）

最后一天做的是硬核但不性感的事：生产化加固。

这一步看起来不“炫”，但决定你这套系统是玩具还是生产工具。

已经完成/落地的关键能力（当前状态快照）

  • OpenClaw 升级到 2026.2.24
  • 网关稳定运行（loopback、本机可达）
  • 定时任务 6 条持续运行，最近状态均为 ok
  • 形成两个可复用脚本：
    • scripts/bootstrap-openclaw.sh（一键恢复部署骨架）
    • scripts/verify-openclaw.sh（一键体检）
  • 内容侧形成“Markdown -> HTML -> 推送/入库”的闭环（Notion API 直写待修 key）

我做的两个脚本，分别干嘛（新手版解释）

1）bootstrap-openclaw.sh

你可以把它理解成：“换机/重装时的一键搭骨架脚本”。

主要做这些事：

  • 准备环境变量文件模板（如 ~/.config/openclaw/env）
  • 配置 systemd 的 EnvironmentFile 引用
  • 确保知识库路径在配置里（memorySearch.extraPaths）
  • 写入常用模型别名（比如 g31p / g3f）
  • 重载并尝试重启 gateway

它不是“万能安装器”，但能把你最容易漏的配置骨架一次性立住。

2）verify-openclaw.sh

你可以把它理解成：“巡检/体检脚本”。

主要检查：

  • OpenClaw 版本
  • Gateway 是否可连、服务是否在跑
  • cron 任务数量和最近状态
  • memory 检索模式（是否还在 fts-only）
  • 关键 skills 是否存在
  • MATON_API_KEY 是否进了 gateway 进程环境
  • workspace 关键文件是否齐全

这个脚本最大的价值是：

  • 你不再靠“感觉好像能用”
  • 而是有一份实际检查结果

新手建议：每次升级后至少跑这三条

openclaw status
openclaw update status
/root/.openclaw/workspace/scripts/verify-openclaw.sh

新手照着走：一份“最短上手路线图”（建议收藏）

如果你今天刚开始，我建议你按下面顺序来，别乱跳：

第 1 步：确认系统活着

openclaw status
openclaw gateway status

看到网关可达、会话正常，就算过关。

第 2 步：先写 3 个文件（人格/用户/长期记忆）

  • SOUL.md
  • USER.md
  • MEMORY.md

先写基础版，后面慢慢补，不用一开始追求完美。

第 3 步：跑一次安全与版本检查

openclaw update status
openclaw security audit --deep

先看懂，不要一上来就到处 --fix。

第 4 步：先跑通一个“真实小任务”

比如：

  • 让它整理一篇文章
  • 保存到 workspace
  • 发到 Telegram 给你预览

只要这一步跑通，你的信心会暴涨。

第 5 步：再补知识库与工作流约束

告诉它：

  • 先查你的知识库
  • 输出怎么标来源
  • 哪些动作要先问你
  • 哪些动作可直接做

这一步决定它后面是不是“像你团队的人”。

第 6 步：最后再做 cron / Notion / 生产化脚本

这时候你已经知道自己在干嘛了，再上自动化才不容易翻车。

新手常见坑（我替你踩过几个）

坑 1：把环境变量写坏了，还以为是接口挂了

我这次就踩到了一个典型坑：MATON_API_KEY 被写成了占位内容（不是可用 key），结果看起来像是 Notion API 有问题。

排查思路应该是：

  1. 先看环境变量有没有真的生效
  2. 再看 gateway 进程里有没有拿到
  3. 再测 API 请求

不是上来就怀疑平台。

坑 2：把“能跑通一次”当成“已经稳定”

一次成功不代表生产可用。

至少补上：

  • 版本检查
  • 安全审计
  • 巡检脚本
  • 日志落盘

不然你下次升级后就会开始“玄学排障”。

坑 3：知识库加进去了，但 embeddings 没开

这时候会出现一个错觉：

  • “我明明把知识库目录配了，为什么还觉得它找得不准？”

因为当前可能只是 fts-only（关键词检索），还不是语义召回。

这不是 OpenClaw 不行，是你还没把 embeddings provider 配上。

下一步（我自己的待办，也适合新手参考）

接下来优先级最高就两件事：

  1. 补 embeddings provider
    • 让记忆检索从“关键词命中”升级到“语义召回”
    • 然后执行索引，验证召回质量
  2. 修好 Notion key 配置
    • 恢复 Notion API 直写稳定性
    • 把“文章选题草稿箱”入库链路彻底跑稳

做完这两步，这套系统基本就能长期跑业务了。

结尾：逮虾户，不是飙车，是稳交付

《逮虾户的春节七天》真正的后劲，不是“我把 OpenClaw 装上了”，而是：

我把它从“会说”调成了“会做、会交付、可复盘”。

如果你是新手，真不用一上来就追求“全自动、全技能、全平台联动”。

先跑通一条小闭环：

能用 → 稳定 → 可复用 → 再自动化。

这条路比你想象的快，也比你想象的稳。

附录A：我的 OpenClaw 关键配置（可复用模板）

下面是我这次实盘跑通后保留下来的“核心配置骨架”，敏感值已脱敏。

1）运行与模型层

  • OpenClaw 版本：2026.2.24
  • 运行方式：gateway + systemd user service
  • Gateway：bind=loopback，端口 18789（仅本机访问，安全性更高）
  • 默认模型：cpa/gpt-5.3-codex
  • 模型别名：
    • g31p -> cpa/gemini-3.1-pro-high
    • g3f -> cpa/gemini-3-flash
  • 并发设置：
    • agents.defaults.maxConcurrent=4
    • agents.defaults.subagents.maxConcurrent=8

2）工作区与记忆层

  • workspace：/root/.openclaw/workspace
  • 记忆检索附加路径：/opt/openclaw/knowledge/secendME
  • 当前记忆模式：fts-only（关键词检索）
  • 当前状态说明：embeddings provider 尚未启用，后续应升级到语义召回

3）消息与渠道层

  • Telegram 通道：已启用并稳定
  • 插件（plugins）已启用：
    • telegram
    • google-gemini-cli-auth
  • 消息策略：群聊以 mention 触发应答为主（减少噪音）

4）配置治理层（我踩坑后的做法）

  • 环境变量统一放：~/.config/openclaw/env
  • 权限固定：chmod 600 ~/.config/openclaw/env
  • systemd drop-in 引用 EnvironmentFile，避免“重启后丢 key”
  • 每次改配置后固定执行：
    • systemctl --user daemon-reload
    • systemctl --user restart openclaw-gateway

附录C：我定的心跳规则（Heartbeat）

1）执行节奏

  • 频率：每 1h
  • 活跃时段：08:30-23:00
  • 默认目标：最近会话（target=last）

2）检查项（固定四类）

  1. AI 热点选题
  2. 政企数字化 / 智能化热点选题
  3. 海外中文新闻（chinanewsx）
  4. 进行中任务进展

3）消息节制规则

  • 没有新变化、没有可执行建议、没有紧急事项：HEARTBEAT_OK
  • 非紧急事项尽量合并成一条消息
  • 夜间（23:00-08:30）默认不主动打扰，除非紧急

4）有事项时的输出格式

  • 【发现】一句话结论
  • 【依据】2-3 条关键点
  • 【建议动作】最多 3 条（按优先级）

附录D：我定的定时任务规则（Cron）

当前线上稳定运行任务共 6 条，最近状态均为 ok：

  1. 郑州晨报0730：30 7 * @ Asia/Shanghai
  2. 海外中文新闻简报0800：0 8 * @ Asia/Shanghai
  3. 海外中文新闻简报1400：0 14 * @ Asia/Shanghai
  4. 股市赛道简报1600：0 16 * @ Asia/Shanghai
  5. healthcheck:update-status：0 4 * @ UTC（每日版本检查）
  6. healthcheck:security-audit：0 4 0 @ UTC（每周安全审计）

我给新手的 cron 原则

  • 先上“少而稳”的 2-4 条，别一口气塞满
  • 每条任务必须有明确产出目标（给谁看、解决什么）
  • 定时任务要能失败可见（日志落盘 + 错误可追）
  • 周期任务优先放在你真的会看的时间段

附录E：一键体检与一键恢复（脚本说明）

scripts/verify-openclaw.sh

作用：一键看“现在是否健康”。

会检查：

  • 版本、gateway、cron、memory 模式
  • 核心技能是否存在
