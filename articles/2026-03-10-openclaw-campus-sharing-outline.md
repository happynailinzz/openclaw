# OpenClaw 分享提纲：从火爆出圈到真正跑起来

作者：余炜勋  
面向对象：有一定技术基础、但第一次上手 OpenClaw 的新手

---

# 一、先讲结论：为什么最近 OpenClaw 这么火

如果用一句话概括 OpenClaw 的热度来源，我的判断是：

**它不是单纯又一个“聊天机器人壳子”，而是在把“大模型 + 工具调用 + 长时记忆 + 自动化执行 + 多渠道触达”真正揉成一个可落地的个人/团队 Agent 工作台。**

过去很多 AI 工具的问题，不是模型不够强，而是：

- 只能聊天，不能真正执行
- 记不住上下文，第二天像失忆
- 能力散，搜索、写作、爬取、发消息、定时任务彼此割裂
- 看起来很酷，但一落地就变成“演示软件”

OpenClaw 这波能火，本质上是因为它把大家真正想要的几件事连起来了：

1. **它是 Agent，不只是对话框**  
   能调工具、能跑任务、能跨会话、能做自动化。

2. **它是可自托管的**  
   对有一定技术基础的人来说，不用把全部工作流押在第三方 SaaS 上。

3. **它有“工作流味”**  
   不只是问答，而是能接入 Notion、消息渠道、浏览器、cron、memory、skills。

4. **它的扩展性很强**  
   skill、plugin、脚本、模型路由、节点能力，都可以按自己的业务场景长出来。

所以我建议大家理解 OpenClaw，不要把它当成“更强一点的 AI 聊天工具”，而要把它看成：

> **一个能把 AI 从“回答问题”推进到“帮你持续做事”的基础设施。**

# 二、我们是怎么部署的

我这边的部署思路，核心是两个字：**官方原版**。

不是自己魔改一大堆底层，不是 fork 出一个完全不同的分支，而是：

- **优先用官方原版 OpenClaw**
- 用 **Trae + Claude Code** 辅助排查、改脚本、补自动化
- 把个性化能力尽量沉淀在：
  - workspace
  - scripts
  - config
  - memory
  - skills

这样做的好处很现实：

1. **升级不容易炸**  
   如果你直接改 OpenClaw 核心代码，后面版本一升级，很容易冲突。

2. **问题边界清楚**  
   是官方链路问题，还是你自己的脚本/配置问题，排查成本会低很多。

3. **更适合新手复现**  
   你给别人分享的时候，大家更容易照着做，而不是照着你的一堆定制 patch 猜。

4. **Trae + Claude Code 更适合做“辅助开发”**  
   例如：
   - 写部署脚本
   - 生成配置模板
   - 修发布链脚本
   - 加巡检/恢复脚本
   - 做 HTML 后处理

也就是说，**OpenClaw 本体尽量官方原版，定制尽量放外层。**  
这是我踩过坑之后，比较坚定的一个建议。

# 三、推荐两套配置方案

## 方案 A：本地 macOS + 国产大模型

### 适合谁
- 想先低门槛上手
- 手边有 Mac mini / MacBook
- 希望日常中文任务更顺手
- 对延迟、隐私、本地可控性更敏感

### 推荐形态
- 机器：macOS（尤其是常开 Mac mini 很适合）
- OpenClaw：官方原版
- 模型：接国产模型或国内兼容 OpenAI 接口的中转
- 渠道：先接 Telegram / 飞书 / 企业微信等常用入口
- 工作区：把知识库、脚本、配置、发布链都放 workspace

### 这套的优点
- 上手快
- 改配置方便
- 调试体验好
- 适合做内容生产、知识库检索、自动化提醒、消息聚合

### 这套的注意点
- 本地网络环境会影响模型调用稳定性
- 常驻运行要注意休眠、网络切换、权限弹窗
- 新手常见问题不是“装不上”，而是“装上了但链路不稳定”

---

## 方案 B：海外服务器 + 国外大模型中转站

### 适合谁
- 想要更稳定的常驻运行
- 想接更多国外模型和生态能力
- 有 VPS / 云主机基础
- 希望任务执行更像真正的服务端系统

### 推荐形态
- 机器：海外 Linux 服务器
- OpenClaw：官方原版 + systemd 守护
- 模型：国外大模型 + 中转站统一管理
- 安全策略：loopback 绑定、环境变量隔离、systemd 管理
- 自动化：heartbeat + cron + scripts

### 这套的优点
- 更适合 24x7 运行
- 模型选择面更广
- 适合做多渠道消息、定时任务、长流程自动化

### 这套的注意点
- 不是装完就完，**安全性和可维护性要求明显更高**
- 必须把密钥治理、网关绑定、服务守护、日志巡检做好
- 新手最大风险不是功能不够，而是“把一套长期在线服务当本地玩具来配”

# 四、部署过程里做了哪些关键配置

我建议把配置理解成 4 层：

## 1）模型层
- 配模型别名
- 区分轻任务、重任务、长任务
- 国内/国外模型按场景分流

## 2）记忆层
- 本地热记忆：`memory/` + `MEMORY.md`
- 路由脚本：`scripts/memory-retrieval-router.py`
- 可扩展到 Memos / Notion / Obsidian 并行检索

## 3）自动化层
- `cron` 做精确定时
- `heartbeat` 做低打扰巡检
- 发布链脚本、巡检脚本、恢复脚本单独沉淀

## 4）工作流层
- 知识库优先级
- 发布链 SOP
- HTML 后处理
- Notion 入库规范
- 消息投递规则

换句话说，真正决定 OpenClaw 能不能跑顺的，不只是“装好了没有”，而是这四层有没有真正接起来。

# 五、包含了哪些插件和 skills

这里先强调一个认知：

**OpenClaw 不是“装一个主程序就完事”的产品，它更像一个 Agent runtime + skills/plugin 生态。**

## 1）Skills
从 CLI 能看出来，OpenClaw 原生就有 skills 体系，适合把一类场景封装成复用能力。

我这边实际工作流里，比较重要的 skill / 能力包括：

- 内容生产相关
  - markdown -> html
  - 发布到微信公众号
  - Notion 入库
- 知识与记忆相关
  - memory retrieval
  - Obsidian / 本地知识库整合
- 自动化与巡检相关
  - cron
  - heartbeat
  - security audit
- 浏览器与网页处理相关
  - browser automation
  - 网页抓取/回退链路
- 平台扩展相关
  - mcporter / notion / 其他外部集成

## 2）Plugins
CLI 里也能看到 OpenClaw 有插件系统，支持：
- 安装
- 启用/禁用
- doctor 检查
- 更新

我的建议是：

> **skills 用来扩展“能力”，plugins 用来扩展“系统级集成”。**

新手不要一上来装一堆，先保留最小可用集，再逐步加。

# 六、它到底能做什么

这是分享里最容易讲嗨，也最容易讲虚的部分。

我建议把 OpenClaw 的能力分成 6 类来讲：

## 1）对话与任务执行
不是只有聊天，还能实际调用工具执行任务。

## 2）知识库与记忆
能接本地知识库、长期记忆、每日记忆，让它不至于每次都重新认识你。

## 3）自动化工作流
能跑 cron、heartbeat、定时提醒、巡检、监控、日报/简报。

## 4）内容生产
能做选题、写作、改稿、HTML 生成、发布链入库。

## 5）多渠道触达
能接消息系统、把结果主动发出去，而不是只在当前对话框里等人来问。

## 6）系统可扩展
有 skills、plugins、脚本、模型路由、browser、node 等扩展能力。

如果给校友群一句最容易理解的话，我会这么说：

> **OpenClaw 最有价值的地方，不是“更聪明地回答问题”，而是“能把回答、执行、记忆、发布和提醒连起来”。**

# 七、我们做了哪些优化

这部分其实最值得讲，因为这里才是真正的“从能跑到好用”。

## 1）把生产流程脚本化
例如：
- `bootstrap-openclaw.sh`
- `verify-openclaw.sh`
- `publish.sh`

意义是：
- 部署可复现
- 巡检可执行
- 发布可标准化

## 2）把记忆做成分层
- 本地记忆保热
- 外部记忆做补充
- 回答前做路由，而不是盲搜

## 3）把 cron 和 heartbeat 分工明确
- 精确定时任务用 cron
- 轻量巡检用 heartbeat
- 成功尽量静默，失败才提醒

## 4）把 HTML 发布链后处理固化
这是非常典型的一个坑：

看起来内容生成了，但如果最后输出的 HTML 风格不统一、排版有脏点、甚至出现单独一行的“。”，那实际发布体验会很差。

所以后面我把这套规则直接固化进脚本：
- 居中下划线主标题
- 蓝色副标题/作者
- 蓝底白字二级标题
- 左侧蓝线引用块
- 自动清理单独一行的标点和碎句

## 5）把生产环境和实验环境意识分开
这是很多人会忽视的。

有些东西可以试，但不能在生产链路里乱试。比如：
- 新 skill
- 新插件
- 新模型路由
- 新消息链路

# 八、我踩过的坑

## 1）不要轻易改 OpenClaw 核心源码
短期看很爽，长期维护很难受。升级时尤其容易炸。

## 2）消息链路最容易“看起来成功，实际上没送达”
尤其是 cron、last channel、默认目标这些地方，必须做显式投递和回执验证。

## 3）HTML 最后的 10% 最折腾
正文写出来不难，难的是最后一层发布体验。很多问题不是内容问题，而是样式、排版、兼容性问题。

## 4）新手最大误区是“先堆能力，再想稳定”
正确顺序应该是：

> **先跑通最小链路，再加能力，再做自动化，最后再谈生产化。**

## 5）密钥治理和服务守护不能后补
特别是服务器部署，必须一开始就把：
- 环境变量
- systemd
- loopback 绑定
- 配置备份
- 安全审计
做好。

# 九、可用性、安全性和稳定性怎么讲

这一部分我建议你分享时不要讲成“大而全安全课”，而是讲成 3 个实操判断。

## 1）可用性
关注的是：**它是不是能持续帮你做事**。

关键点：
- 任务链路是否跑通
- 关键消息能否送达
- 发布链是否闭环
- 失败后有没有回退方案

## 2）安全性
关注的是：**你有没有把它当成长期在线系统来管理**。

关键点：
- 密钥不明文散落
- gateway 不裸奔暴露
- 最好 loopback 绑定
- 做安全审计
- 插件/skills 不要乱装

CLI 里其实也已经提供了：
- `openclaw security audit`
- `openclaw plugins doctor`

这些都值得新手上手后尽早跑一遍。

## 3）稳定性
关注的是：**出问题时你是不是能定位、恢复、回滚。**

关键点：
- systemd 守护
- 巡检脚本
- 配置备份
- cron/heartbeat 分工
- 日志与回执

一句话总结就是：

> **OpenClaw 能不能长期用，不取决于你装了多少功能，而取决于你有没有把它当成一套系统来运营。**

# 十、我给新手的上手建议

如果面对有一定 IT 基础、但第一次上手的小白，我会给 5 条建议：

1. **先用官方原版，不要一开始就魔改。**
2. **先跑通最小闭环：模型调用 + 一个消息入口 + 一个简单任务。**
3. **先确定一套主场景，再扩展 skills/plugins。**
4. **从第一天就做密钥治理、备份和巡检。**
5. **把踩坑经验写成脚本和 SOP，不要靠脑子记。**

# 十一、适合分享时收尾的一句话

如果让我用一句话做结尾，我会这么说：

> **OpenClaw 真正吸引人的，不是它“会说话”，而是它开始像一个能长期协作的 AI 同事。**

但前提是，你不能只把它当玩具装起来，而要把它当一套要长期运行、持续进化、可维护的系统去建设。
