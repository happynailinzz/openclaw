## qmd 可用性体检与路由修复
- 对 qmd 做了分层排查：安装层可用，索引层可用但更新时间偏旧，路由层存在 CLI 参数与输出解析不兼容问题，召回质量仍以关键词检索为主。
- 关键结论：此前 `scripts/memory-retrieval-router.py` 调用 `qmd search` 时使用了错误参数 `--limit`，而当前 qmd CLI 应使用 `-n`；同时脚本对 qmd 输出头部解析不兼容，导致经常出现 `qmd_hits.ok=true` 但 `hits=[]` 的假可用状态。
- 已修复 `scripts/memory-retrieval-router.py`：
  - `qmd search <query> --limit <n>` -> `qmd search <query> -n <n>`
  - 调整 qmd 输出头部解析，兼容 `qmd://...:line #id` 格式
  - 仅在命中块内提取 `Title/Score/Context/snippet`
- 修复后回归验证：
  - 查询“源网荷储 数字化”可稳定拿到 qmd 命中 1 条
  - 查询“余炜勋”可稳定拿到 qmd 命中 3 条
  - 查询“记忆路由”“郑州 数字化 升级”仍无 qmd 命中，说明当前更大的剩余问题是索引覆盖与关键词召回边界，不再是路由 bug
- 额外发现：`qmd query` 仍会提示 `1406 documents (99%) need embeddings`，并触发约 1.28GB 模型下载/CPU fallback；说明语义检索层尚未稳定落地，当前不宜把 qmd 当成成熟的语义召回主通道。
- 已提交 git commit：`fe217b7` - `fix qmd router cli parsing`
