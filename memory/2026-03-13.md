## OpenClaw 最近更新中 OpenAI / GPT-5.4 / Codex 优化结论（2026-03-13 10:46 UTC）

### 用户关注点
- 用户主动询问：最近几次 OpenClaw 更新对 OpenAI 模型做了哪些优化，以及在使用 `GPT-5.4` 和 `codex` 时应如何优化使用。
- 这说明用户当前对 OpenClaw 内的 OpenAI 系模型，尤其是 `GPT-5.4` 与 `codex`，有较高实际使用频率。

### 已总结的关键版本结论
- 最新一版重点增强了 `OpenAI/GPT-5.4 fast mode`：已支持 session 级 `/fast` 开关，并在 TUI、Control UI、ACP 中统一接入，同时会针对 OpenAI/Codex 做请求整形。
- 同版还加入了 OpenAI/Codex fast-mode tier 的实时校验，说明 fast mode 不只是 UI 层开关，而是已开始落实到请求层。
- 同版修复了 `gpt-5.3-codex-spark` 在 `openai-codex/*` 路径上的兼容性，避免继续走 OpenAI 已拒绝的旧 `openai/*` Spark 直连方式。
- 2 月 27 日附近版本把 `openai-codex` 默认传输改成 `WebSocket-first`，保留 SSE fallback，提升了交互式使用的实时性与稳定性。
- 2 月 18 日附近版本增加了 `OpenAI Codex CLI auth provider integration`，说明 Codex 接入链路正在被做成更原生的认证路径。

### 对用户的实用建议口径（已输出）
- 默认分工应为：`GPT-5.4` 负责“想清楚再说”的任务，如方案设计、复杂分析、代码评审、长链路决策；`codex` 负责“直接下手”的任务，如改代码、补测试、做局部重构、生成 patch。
- 最优工作流应为：先让 `GPT-5.4` 做任务拆解和验收标准定义，再把明确任务交给 `codex` 执行，最后再由 `GPT-5.4` 做复核。
- 对 `codex` 的稳定使用原则：任务颗粒度要小、上下文只给相关内容、验收标准前置、输出格式固定（改了什么/哪些文件/风险点/验证步骤）。
- 对 `GPT-5.4` 的高价值使用原则：优先用于方案比较、风险识别、把模糊需求改写成适合 coding agent 执行的任务说明，而不是浪费在机械代码生成上。
- 默认判断规则：如果任务里还存在“到底要做什么”的不确定性，用 `GPT-5.4`；如果任务已经变成“按标准把东西做出来”，用 `codex`。
