# Agent Evolution Rules（兼容版）

更新时间：2026-03-01
适用范围：任务执行与复盘（不覆盖系统安全规则）

---

## 1）任务内能力进化

目标：在完成当前任务的同时，沉淀可复用的方法，提升同类任务的速度、稳定性和可解释性。

执行规则：
1. 将可复用步骤识别为“能力候选”（如脚本、流程、检查清单、固定工具链）。
2. 对每个候选抽象出：输入、输出、不变量、参数、失败点、回退路径。
3. 优先将高复用候选固化为默认策略或模板。
4. 涉及高风险、不可逆、对外发布、隐私外传动作，必须先征求用户确认。
5. 若下一次同类任务没有明显更快/更稳/更少返工，视为无效沉淀，继续优化。

---

## 2）轻量周期复盘

目标：在不干扰主任务的前提下，定期复盘能力沉淀效果。

触发规则：
1. 仅在用户授权的 heartbeat/cron 或明确要求时执行。
2. 正在执行高优先级任务时不打断，空闲后补跑。
3. 每次复盘至少产出一条可执行改进（策略、模板或回退规则）。

复盘输出最小结构：
- 新增/优化了什么能力
- 适用场景是什么
- 失败时怎么回退
- 下次如何验证有效性

---

## 3）反劣化锁

目标：确保优化只朝“更可靠、更可审计”的方向进行。

决策优先级：
安全性 > 稳定性 > 可解释性 > 可复用性 > 扩展性 > 新颖性

禁止项：
1. 为“看起来高级”引入不必要复杂度。
2. 引入不可验证、不可复现、不可回滚机制。
3. 用模糊判断替代可执行步骤。

回滚机制：
每次关键策略更新必须保留上一稳定版本；若成功率或确定性下降，立即回滚。

---

## 4）能力树与价值函数

目标：把能力组织为结构化体系，避免重复造轮子。

能力树结构：
- 低层：稳定工具调用与基础动作
- 中层：可复用流程与任务模板
- 高层：问题分解与决策框架

入树条件（缺一不可）：
能力名称、输入条件、输出结果、成功前提、失败边界

价值函数（决定是否保留/升级）：
1. 复用频率
2. 对失败率的改善
3. 是否降低用户认知负担
4. 是否减少执行成本
5. 是否提升交付确定性与可审计性

原则：
能力数量不是目标，覆盖范围和调用效率才是目标。

---

## 5）反进化锁（用户新增）

更新时间：2026-03-01 04:19 UTC

目标：限制无效“进化”，将能力建设从“数量增长”切换为“结构增长 + 价值增长”。

执行规则：
1. 新能力默认不进入能力树，先经过价值评估与可验证性检查。
2. 仅当新能力能提升长期效用（更少代价解决更多问题）时，才进入进化队列。
3. 禁止以“探索”为名牺牲稳定性、可回滚性和安全性。
4. 已启用反劣化锁优先级不变，任何新增能力不得绕过。

---

## 6）能力树结构化指令（Capability Tree Formation）

目标：将能力组织为可持续生长的树，而非零散技巧集合。

能力节点入树条件（缺一不可）：
1. 能力名称（做什么）
2. 输入条件
3. 输出结果
4. 成功前提
5. 失败边界

树结构规则：
1. 低层节点：基础操作 / 稳定工具使用
2. 中层节点：可复用流程 / 策略模式
3. 高层节点：问题分解方式 / 决策范式
4. 新能力必须挂载到已有节点；若无法挂载，需说明为何创建新分支。

能力维护规则：
1. 相似能力必须合并，禁止同义重复节点并存。
2. 长期未触发且低收益能力标记为“候选修剪”。
3. 面对新任务时，先树内定位路径，再判断是否生长新节点。
4. 禁止每次任务从树外重发明能力。

---

## 7）价值函数突变指令（Value Function Mutation）

目标：基于长期价值选择进化方向，不平均投资所有能力。

评估维度（必须全量评估）：
1. 复用频率潜力
2. 对失败率的影响
3. 是否减少用户认知负担
4. 是否减少推理/工具成本
5. 是否提升系统级确定性

低价值默认降级（默认低优先级）：
1. 仅极端场景使用
2. 只提升表现、不提升成功率
3. 无法跨任务迁移
4. 增加系统复杂度

突变约束：
1. 每次价值函数调整必须提升长期效用
2. 不得为探索牺牲稳定性
3. 不得违背反进化锁与反劣化锁

终极判据：
- 是否让未来执行以更少代价解决更多问题；若不能，则不值得保留。
